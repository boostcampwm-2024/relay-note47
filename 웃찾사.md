            - 코멘트 < 테스트케이스 유용
        - 짧은 코드
            - 함축적 코드 < 명시적 코드

    - 유연성
        - 유연성이 있는 코드가 더 좋겠지만
        - 유연성 있고  + 어려운 코드 < 유연성 없고 + 쉬운코드
- 좋은 코드 < 잘 동작하는 코드
    
    ⇒ Make it work ⇒ Make it Better
    
    ⇒ 기술부채가 쌓이게 됨

### S041 양승현 - [ 개발자원칙 8장. 제어할 수 없는 것에 의존하지 않기 ]

> “ 일정을 지키고자 버그가 많은 소프트웨어를 출시하는 것이 마음에 들지 않습니다. 어떻게 하면 일정을 연기해서 안정된 소프트웨어를 내는 것이 중요하다고 리더를 설득할 수 있을까요? ”
> 

 

“프로그래머에게 요구되는 것은 100점이 아닌 80~90점 짜리 프로그램을 기한 내에 완성하는 일이다. 퀄리티를 중요시하지 않는게 아니라, 어떻게 80~90점 짜리 소프트웨어를 개발할 수 있을지에 초점을 두어야 한다.”

- 인프런(CTO)이신 향로님은 위와 같은 질문이 왔을 때 나카지마 사토시님의 이야기를 항상 전달한다고 합니다.

타이트한 일정속에서도 이런 80~90%의 일정을 소화하는 분들의 특징은 A 코드 B코드 둘 중 뭐가 좋을지에 대한 고민 과정에서 시간 소모보단, 더 적합한 코드를 판별하는 자신만의 기준과 원칙을 적용해 곧바로 선택해 나간다는 것입니다. (e.g. DRY, YANGI, KISS)

- 저는 차후에 발생할 수 경우들을 고려해 항상 100를 원했었는데.. 점수가 높은, 저만의 기준, 원칙을 적용해서 프로그램을 기한 내 완성할 수 있는 자세를 갖을 수 있도록 더 많은 코드를 보며 탐구해나가고, 경험을 쌓아야 겠다고 생각했습니다.
- 또한 제어할 수없는 것에 의존하지 말아야한다는 것이 코드속에서도 적용되고, 현실에서도 적용될 수 있다는 것을 알게 되었습니다.
- “ 이미 발생한 사건, 결정권이 없는 일들은 제어 할 수 없다. 이들에 의존하지 말고 제어할 수 있는 것들에만 의존 해야한다. “ 라는 말씀을 하셨는데, 이 마음가짐을 본받아야겠습니다: )
